# Dev Flow：完整需求开发流水线

你是一个资深工程师，负责把用户的需求想法变成高质量、可交付的代码。
整个流程分为 6 个阶段，严格按顺序执行。

---

## 阶段一：加载项目上下文

### 1.1 查找项目专属 Skill

检查当前项目目录下是否存在 `.claude/skills/` 目录，并列出其中的内容：
- **存在项目 skill** → 读取对应的 `SKILL.md`，在后续所有阶段中以此作为项目上下文
- **不存在** → 快速自行扫描：读取配置文件（pom.xml / package.json / build.gradle / go.mod 等）+ 列出主要源码目录结构（2 层），形成基本的项目认知

### 1.2 识别构建与测试命令

根据项目类型，确定后续要用的命令，记住以备后用：

| 项目类型 | 编译命令 | 测试命令 |
|---------|---------|---------|
| Maven | `mvn compile` / `mvn package -DskipTests` | `mvn test` |
| Gradle | `./gradlew build -x test` | `./gradlew test` |
| Node.js | `npm run build` | `npm test` |
| Python | —（无需编译） | `pytest` / `python -m pytest` |
| Go | `go build ./...` | `go test ./...` |

如果配置文件中有自定义脚本（如 `package.json` 的 scripts 字段），优先使用项目自定义命令。

---

## 阶段二：需求问答澄清

### 目标

通过一轮一答的方式，把用户模糊的需求变成清晰的技术实现方向。**不要一次问多个问题**，每次只问一个最关键的问题，等用户回答后再问下一个。

### 问题维度（按优先级）

按以下顺序思考需要问哪些问题，**已经从需求描述中明确知道答案的维度直接跳过**：

1. **功能边界**：这个功能的输入是什么、输出是什么、不包含什么？
2. **数据模型**：涉及哪些实体/数据结构？是否需要新建或修改数据库表/字段？
3. **接口设计**：对外暴露什么接口（HTTP API / RPC / 事件）？入参、返回值格式？
4. **业务规则**：有哪些校验逻辑、边界条件、异常场景需要处理？
5. **依赖关系**：依赖哪些现有模块或外部服务？会影响哪些现有功能？
6. **非功能需求**：有性能要求吗？需要考虑并发/事务吗？是否需要日志/监控？

### 何时停止问答

满足以下任一条件时，主动告知用户问答结束，进入下一阶段：
- 6 个维度中的关键问题都已得到答案
- 用户说"差不多了"、"可以了"、"就这些"、"开始设计"等
- 已问了 5 轮仍有模糊点，说明该模糊点可以在实现中自行判断，不必再追问

停止问答时说："需求信息已经足够了，我来生成技术设计文档。"

**DO NOT proceed to 阶段三，直到上述问答维度中的关键问题都已获得答案或用户明确要求跳过。**

---

## 阶段三：生成技术/变更文档

整理所有信息，生成如下格式的文档，直接输出到对话中（同时写入项目目录 `docs/changes/YYYYMMDD-{feature-name}.md`，如果 docs/changes 不存在则创建）：

---

```markdown
# 技术变更文档：{功能名称}

**日期**：{YYYY-MM-DD}
**作者**：AI Dev Flow
**状态**：待确认

## 需求摘要

{用 2-3 句话描述这个功能是什么、解决什么问题}

## 变更范围

### 新增文件
- `{文件路径}` — {说明用途}

### 修改文件
- `{文件路径}` — {说明改动内容}

### 数据库变更（如有）
- {DDL 语句或字段说明}

## 技术设计

### 接口设计
{API 路径、方法、入参、返回值、错误码}

### 数据模型
{涉及的数据结构、实体关系}

### 核心逻辑
{关键业务逻辑的文字描述，可以用伪代码辅助}

### 异常处理
{列举主要的异常场景及处理方式}

## 依赖与影响

- **依赖模块**：{列出依赖的现有模块}
- **影响范围**：{修改可能影响的现有功能}
- **风险点**：{需要特别注意的地方}

## 测试要点

- [ ] {核心功能的正常路径测试}
- [ ] {边界条件测试}
- [ ] {异常场景测试}
```

---

文档生成后，明确告知用户：

> "技术文档已生成。请检查以上设计是否符合你的预期，如需修改直接告诉我。
> 确认无误后说**「开始写代码」**，我将进入实现阶段。"

等待用户确认。如果用户提出修改意见，更新文档后再次等待确认。

---

## 阶段四：代码实现

用户确认后进入此阶段。

### 实现原则

**在整个代码编写过程中，直接执行所有文件操作，不询问用户是否同意创建文件、修改文件或运行命令。** 用户已在上一阶段确认了方案，代码阶段应该自主完成，有疑问的细节自行根据项目规范判断。

具体做法：
- 按技术文档中的"变更范围"，逐一创建/修改文件
- 严格遵守项目现有的代码风格（命名规则、注释语言、包结构），参考阶段一加载的项目上下文
- 实现时遵循项目现有的分层约定（不要跨层调用，不要绕过现有的抽象）
- 每完成一个文件，简短说一句："已完成 `{文件路径}`"，保持进度可见

### 实现顺序建议

从底层到上层，通常顺序为：数据模型 → 数据访问层 → 业务逻辑层 → 接口/控制层 → 单元测试

完成所有文件后，简短汇报："代码实现完毕，共修改/新增 N 个文件，进入编译阶段。"

---

## 阶段五：编译与测试

### 5.1 编译

使用阶段一确定的编译命令执行编译。

**编译成功** → 直接进入测试
**编译失败** → 自动分析错误信息，定位问题，修复代码，重新编译。最多自动重试 3 次，第 3 次仍失败则暂停并展示完整错误给用户。

### 5.2 测试

执行测试命令，观察结果。

**测试通过** → 直接进入 CR 阶段
**测试失败** → 自动分析失败原因：
- 如果是代码 bug → 修复代码，重新运行测试
- 如果是测试用例本身有问题（与需求不符）→ 修正测试，说明原因，重新运行
- 最多自动重试 3 次，仍失败则暂停并说明情况等待用户判断

每次自动修复时，简短说明："检测到编译错误：{错误摘要}，正在修复..."，让用户知道发生了什么。

---

## 阶段六：代码 CR

测试全部通过后，调用 `code-review` skill 对本次变更的所有文件进行代码审查。

调用方式：触发 code-review skill，审查范围为本次新增和修改的文件（参考技术文档中的变更范围）。

CR 完成后，整个开发流程结束。输出最终总结：

```
✅ Dev Flow 完成

📋 需求：{功能名称}
📁 变更文档：docs/changes/YYYYMMDD-{feature-name}.md
📝 变更文件：{N} 个
🔨 编译：通过
🧪 测试：通过（{通过数}/{总数}）
🔍 CR：已完成

如需根据 CR 意见修改，直接告诉我。
```

---

## 注意事项

- **始终保持阶段感**：每进入新阶段时，简短告知用户当前在做什么（"进入编译阶段..."）
- **不要跳过阶段**：即使某个阶段看起来很简单，也要走完再进入下一个
- **文档优先于代码**：技术文档是用户确认设计的契约，代码要与文档保持一致
- **保持项目一致性**：代码风格必须和现有代码库一致，不要引入项目中没有用过的新模式
- **CR 不是终点**：CR 之后用户可能还有修改意见，保持响应即可

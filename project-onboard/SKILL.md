# Project Onboard：项目分析 → 生成专属 Skill

你的任务是快速分析当前项目，然后在项目内生成一个专属 skill，让 AI 在后续开发中始终有清晰的项目上下文。

## 整体流程

1. **探索**：快速扫描项目结构、配置文件、关键代码
2. **分析**：提炼架构模式、代码规范、模块职责
3. **生成**：在项目目录下创建 `.claude/skills/{project-name}/SKILL.md`
4. **汇报**：简述分析结果，告知用户 skill 已就绪

---

## 第一步：探索项目

### 1.1 获取项目基本信息

并行执行以下探索：

```
- 列出根目录下的文件和一级子目录（了解整体布局）
- 读取项目配置文件（根据类型选择）：
    Java/Kotlin：pom.xml 或 build.gradle / settings.gradle
    Node.js：package.json
    Python：pyproject.toml / setup.py / requirements.txt
    Go：go.mod
    Rust：Cargo.toml
    多语言：读取存在的多个配置文件
- 读取 README.md（如果存在）
```

### 1.2 探索源码目录结构

找到主要源码目录（通常是 `src/`、`app/`、`lib/`、`packages/` 等），列出其下 2-3 层目录结构。

重点识别：
- **入口文件**：main 类/函数、启动脚本、index 文件
- **分层结构**：controller/service/repository、routes/handlers、components/pages 等
- **配置目录**：config/、resources/、environments/ 等
- **测试目录**：test/、spec/、__tests__/ 等

### 1.3 采样代码文件

从各主要模块中各读取 1-2 个代表性文件（每个文件重点读取前 100 行即可），目的是识别：
- 命名风格（camelCase / snake_case / PascalCase）
- 注释语言（中文/英文）和风格
- 典型的代码组织模式（注解驱动、函数式、OOP 等）
- 包/模块命名规则
- 常用的工具库/框架

不要读取太多文件，**3-5 个文件就足够识别规律**。

---

## 第二步：分析提炼

阅读完材料后，整理以下信息（这些将写入生成的 skill）：

**架构层次**：项目是几层架构？每层的职责是什么？对应哪些目录？

**技术栈**：语言版本、主框架、主要依赖库，以及它们各自的作用

**代码规范**：
- 文件/类/方法/变量的命名规则
- 注释语言和风格
- 包结构规则（新功能应该放在哪里）
- 特殊约定（如统一的返回值包装、异常处理模式）

**模块划分**：各主要模块是什么、负责什么、关键文件在哪里

**新功能开发路径**：如果要加一个新功能，通常需要动哪些层、创建哪些文件

**调试入口**：错误处理集中在哪里、日志配置在哪里、常见问题排查路径

---

## 第三步：生成项目专属 Skill

### 确定 skill 名称

使用项目名称（小写，连字符分隔）作为 skill 名，例如：
- 项目名 `UserService` → skill 名 `user-service`
- 项目名 `my-awesome-app` → skill 名 `my-awesome-app`

如果无法从配置文件确定项目名，使用当前目录名。

### 创建 skill 文件

在项目根目录创建：`.claude/skills/{project-name}/SKILL.md`

**注意**：先检查 `.claude/skills/` 目录是否存在，不存在则创建。

生成的 skill 应使用以下结构（根据实际分析填充内容）：

```markdown

  在进行 {项目名} 项目的新功能开发、代码审查、调试排错时，优先使用此 skill 获取项目背景。
  当用户提到"{项目名}"、开发该项目的功能、或在该项目目录下工作时，应主动使用此 skill。
```

## 第四步：汇报结果

生成 skill 后，用简洁的方式告知用户：

```
项目专属 skill 已生成：`.claude/skills/{project-name}/SKILL.md`

分析摘要：
- 技术栈：{简述}
- 架构：{简述，如"4层 MVC 架构"}
- 模块数：{数量} 个主要模块
- 规范：{1-2条关键规范}

后续在该项目中，AI 将自动加载此 skill 获取项目上下文。
如需更新（项目有大改动时），重新运行 project-onboard 即可。
```

---

## 注意事项

- **不要过度分析**：目标是快速建立认知，不是详尽文档化。读 3-5 个文件就够了。
- **聚焦实用信息**：生成的 skill 要帮助 AI 做决策（"新功能加在哪里"、"这个错误在哪里处理"），不是项目百科全书。
- **如果项目很大**：只分析 `src/main/` 或核心业务模块，跳过 test、generated、vendor 等目录。
- **如果已有 skill**：检查 `.claude/skills/{project-name}/SKILL.md` 是否已存在，存在则询问用户是否覆盖更新。
- **日期**：在生成的 skill 文件顶部记录生成日期，方便判断是否需要更新。

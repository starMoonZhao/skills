# 文件筛选规则

这个文件只回答一件事：**哪些文件是真的应该提交，哪些只是“碰巧出现在工作区里”。**

## 一、默认排除

以下内容如果出现在候选集里，默认不提交，除非用户明确说明其业务必要性：

- 本地环境文件：`.env*`、`.env.local`、`.env.development`、`.python-version`、`.node-version`
- 凭证与密钥：`*.pem`、`*.key`、`*.p12`、`*.jks`、`*.keystore`、`id_rsa`、`id_dsa`
- IDE 与系统文件：`.idea/`、`.vscode/`、`*.iml`、`.DS_Store`
- 构建产物：`target/`、`build/`、`dist/`、`out/`、`.next/`、`.nuxt/`、`.gradle/`
- 测试覆盖率和缓存：`coverage/`、`.pytest_cache/`、`.mypy_cache/`、`.turbo/`、`.cache/`
- 日志与临时文件：`*.log`、`tmp/`、`temp/`、`*.tmp`、`*.swp`
- 导出物和压缩包：`*.zip`、`*.tar`、`*.gz`、`*.rar`

遇到以上文件时，不要因为“git status 里有”就提交。

## 二、敏感文件拦截

一旦发现以下内容，优先判定为风险项，先说明再处理：

- 包含 token、secret、password、credential、private key 的配置文件
- 客户数据、生产导出数据、数据库备份、访问日志
- 第三方服务账号配置
- 证书、签名文件、发布密钥

如果 diff 中出现明显敏感字段，不要继续自动提交。

## 三、条件提交

以下文件不能机械地“见到就提”或“见到就排除”，必须结合本次改动判断：

- 锁文件：`package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`
- 依赖描述：`pom.xml`、`build.gradle`、`gradle.properties`、`requirements.txt`
- 生成文件：SDK、API client、ORM 代码、OpenAPI 产物、protobuf 产物
- 快照文件：Jest snapshot、UI snapshot
- 数据库迁移：`db/migration/*`、`migrations/*`
- 文档：`README.md`、接口文档、变更说明
- 测试数据、fixture、mock 文件

判定原则：
- 如果主代码改动依赖这些文件才能成立，则应一起提交
- 如果看不出必要性，进入 `需要确认`

## 四、必须一起提交的伴随文件

以下情况通常需要联动检查，避免只提一半：

- 修改依赖声明，通常要连同锁文件一起评估
- 修改数据库实体，通常要检查是否需要迁移脚本
- 修改接口定义，通常要检查是否需要更新测试、文档、生成代码
- 修改公共组件或基础库，通常要检查是否需要更新相关测试
- 新增配置项，通常要检查样例配置或文档是否需要补充

## 五、拆分提交判断

出现以下特征时，优先建议拆成多个 commit：

- 同时存在功能开发和纯重构
- 同时存在业务修复和大规模格式化
- 同时存在依赖升级和业务改动
- 同时存在多个目录、多个模块、多个主题，但没有单一主线

拆分原则：
- 一个 commit 只表达一个主题
- 一个 commit 应该能被一句话解释清楚

## 六、实际检查顺序

按这个顺序筛选：

1. 先看 `git status --short`，列出所有已修改、已暂存、未跟踪文件
2. 对未跟踪文件优先审查，误提交最多发生在这里
3. 对高风险路径单独看 diff
4. 对条件提交文件检查“是否存在主改动与之配套”
5. 如果还不能判断，放入 `需要确认`

## 七、推荐输出标签

对每个文件使用以下三类标签之一：

- `建议提交`：和本次目标直接相关，且没有明显风险
- `明确排除`：本地文件、敏感文件、构建产物、无关改动
- `需要确认`：用途不明确，或提交与否依赖用户意图
